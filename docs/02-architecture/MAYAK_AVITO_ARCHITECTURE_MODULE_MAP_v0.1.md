# Маяк Авито — архитектура и карта модулей

**Версия:** 0.1 (черновик для согласования)  
**Дата:** 2026-07-06  
**Статус:** архитектурная карта; не является детальным ТЗ конкретного модуля  
**Основание:** `MAYAK_AVITO_TARGET_MODEL_v0.1.md`  
**Технический референс Avito:** `Duff89/parser_avito` — только для решений внутри Avito parser adapter.

---

## 1. Главный вывод

Целевая архитектура содержит **13 самостоятельных логических модулей** и **3 сквозных архитектурных контура**.

Это **не 13 микросервисов**. На старте система должна быть модульным монолитом с отдельными воркерами и агентом маршрутизации. Модули нужны для:

- отдельных ТЗ и roadmap-документов для агентов;
- независимой разработки и ревью;
- запрета прямого доступа к чужим внутренностям;
- будущего выделения тяжёлых частей без переписывания доменной модели.

### 1.1. Почему не меньше

Объединять Telegram/MAX с бизнес-логикой, парсер с планировщиком, аккаунты с тарифами или уведомления с ботами было бы быстрее в начале, но сломает параллельную работу и будущий сайт.

### 1.2. Почему не больше

Разбивать систему на отдельные модули для каждой таблицы, кнопки или одного вида фильтра нельзя. Это создаст документационный шум, постоянные зависимости и искусственные границы.

---

## 2. Реальное развёртывание на ранних этапах

На сервере первоначально достаточно пяти исполняемых компонентов:

```text
1. API / modular monolith
2. Worker process (очереди, запуск Маяков, outbox)
3. Scheduler process
4. Windows Egress Agent дома
5. PostgreSQL + очередь/кэш + объектное хранилище при необходимости
```

Telegram, MAX, админка и будущий сайт не должны быть отдельными базами. Они входят в API-процесс как адаптеры/контроллеры одного backend.

Позже отдельно могут быть вынесены parser workers, notification workers, analytics и web frontend — только при реальной нагрузке или операционной необходимости.

## 4. Три сквозных контура

Они не должны быть «ничьими»; за них нужен отдельный владелец ревью. Но это не самостоятельные предметные модули, потому что они пронизывают все модули.

### A. Security & Privacy

- секреты, парольные хеши, одноразовые коды;
- проверка webhook и Mini App payload;
- права ролей на сервере;
- audit log;
- маскирование секретов;
- защита от повторов и подмены внешних событий.

### B. Observability & Operations

- структурные логи, метрики, алерты;
- статусы очередей, маршрутов, Windows-агента;
- резервное копирование;
- диагностика ошибок Маяка;
- release/rollback дисциплина.

### C. Data & Compatibility Governance

- миграции PostgreSQL;
- версионирование API и конфигурации Маяка;
- идемпотентность;
- backward compatibility;
- правила изменения общих контрактов.

## 5. Зависимости: кто от кого зависит

```text
Identity & Access ──────────────┐
                                ├── Beacon Management ──┐
Entitlements & Billing ─────────┘                       │
                                                        ├── Scan Orchestration ── Notification Delivery ── Telegram
Avito Parser Adapter ───── Egress Routing ──────────────┘                                      └──────────── MAX

Admin & Support ───── читает/управляет через публичные сервисы всех модулей
Web Cabinet ────────── использует те же публичные сервисы, что Telegram/MAX
Filter Catalog & Builder ── расширяет Beacon Management + Parser Adapter, не создаёт второй путь данных
```

### Критически важное правило

Ни один внешний интерфейс не пишет напрямую в таблицы другого модуля:

```text
Telegram/MAX/Web/Admin
        ↓
публичные команды/сервисы модулей
        ↓
внутренние таблицы и очередь модуля
```

Например, Telegram не создаёт строку `beacons` напрямую; он вызывает команду создания Маяка. Parser не отправляет сообщения в Telegram напрямую; он создаёт событие, которое передаётся в Notification Delivery.

## 6. Владение данными

| Область данных | Единственный владелец |
|---|---|
| `accounts`, `account_identities`, `credentials`, `roles`, auth challenges | Identity & Access |
| `tariffs`, `entitlements`, `subscriptions`, payments | Entitlements & Billing |
| `beacons`, revisions, source URLs, snapshots, overrides | Beacon Management |
| runs, observations, listing state, baseline/diff | Scan Orchestration & Listing State |
| извлечение `searchCore`, `context`, параметры выдачи, нормализация карточек | Avito Parser Adapter |
| routes, agents, leases, heartbeat | Egress Routing |
| events, outbox, deliveries, template model | Notification Delivery |
| provider-specific inbound event IDs, mapping UI state | Telegram/MAX adapters, но не бизнес-данные |
| admin-only views, support work items | Admin & Support |
| web session/UI presentation state | Web Cabinet |
| definitions/options доказанных фильтров | Filter Catalog & Builder |

Отдельный модуль может читать чужие данные только через контракт или read-model, а не менять чужую таблицу.

## 7. Как это связано с агентными ТЗ

Для каждого из 13 модулей создаётся один долговечный документ:

```text
/docs/modules/<module>/MODULE_SPEC_AND_ROADMAP.md
```

Он прикладывается агенту через Business Bridge периодически. В документе обязательно:

1. назначение и границы;
2. подтверждённые решения;
3. открытые вопросы;
4. ссылки на нужные общие контракты;
5. перечень данных, которыми модуль владеет;
6. разрешённые и запрещённые изменения;
7. roadmap подзадач;
8. тесты и критерии приёмки;
9. формат отчёта и handoff;
10. append-only журнал изменений.

Конкретная задача агента идёт отдельным `TASK-xxx.md` и ссылается на этот модульный документ. В каждом рабочем prompt остаётся краткий guardrail; полный модульный файл прикладывается периодически.

## 8. Очерёдность без привязки к людям и календарю

Это не график разработчиков. Это порядок зависимостей, чтобы любой доступный агент мог брать готовую задачу.

### Волна 1: общая основа

1. Platform & Contracts
2. Identity & Access — минимальный контур аккаунта/ролей/идентичностей
3. Entitlements & Billing — только тарифы и лимиты, без платёжного провайдера
4. Beacon Management — модель и версии конфигурации

### Волна 2: рабочее ядро Маяка

5. Egress Routing — контракт маршрутов и Windows-agent
6. Avito Parser Adapter — только по подтверждённому референсу и реальным ссылкам
7. Scan Orchestration & Listing State — baseline, diff и run states
8. Notification Delivery — outbox и общий формат уведомлений

### Волна 3: bot-first интерфейсы

9. Telegram Adapter
10. MAX Adapter
11. Admin & Support

### Волна 4: последующие продуктовые интерфейсы

12. Web Cabinet
13. Filter Catalog & Builder
14. Платёжный провайдер реализуется как подмодуль Entitlements & Billing, а не отдельная новая предметная система.

## 9. Что можно делать параллельно

После зафиксированных контрактов:

| Параллельная работа | Почему безопасно |
|---|---|
| Telegram Adapter + MAX Adapter | оба используют один Command/API contract, не меняют ядро |
| Avito Parser Adapter + Egress Routing | parser использует fake route; routing тестируется отдельно |
| Notification Delivery + Scan Orchestration | orchestration публикует событие по контракту; delivery читает его |
| Admin & Support + Bot adapters | все используют read/API contracts, не владеют доменными таблицами |
| Web Cabinet + Filter Catalog & Builder | поздние интерфейсы, если контракты Beacon Management уже стабильны |

### Что нельзя делать без совместного решения

- менять `account_id` и правила связывания identity;
- менять модель Маяка и версионирование конфигурации;
- менять тарифные лимиты и семантику active beacon;
- менять формат внешних событий/parser result;
- менять миграции общей схемы;
- менять семантику baseline и ID+цена;
- менять правила авторизации/ролей;
- менять границы данных, которыми владеет другой модуль.

## 10. Критическое решение: не микросервисы на старте

Архитектура должна быть **modular monolith first**:

- один репозиторий;
- одна основная PostgreSQL-база;
- внутренние модули с явными интерфейсами;
- отдельные процессы для scheduler/worker;
- отдельный Windows Egress Agent;
- один CI/CD pipeline;
- один журнал миграций.

Причина: команда из 2–3 нерегулярно работающих агентов/разработчиков не должна нести операционную нагрузку на 13 независимых сервисов, очередей, репозиториев, релизов и сетевых контрактов.

Разделять на микросервисы можно позднее, когда будут доказанные причины: нагрузка парсинга, независимое масштабирование delivery, изоляция рискового egress или операционные требования.

## 12. Критерий готовности архитектурной карты

Архитектура готова к детальному ТЗ модулей, когда согласовано:

- число и границы 13 модулей;
- модульный монолит как стартовый способ развёртывания;
- три сквозных контура;
- владение данными;
- запрет прямых записей между модулями;
- порядок волн и независимые зоны;
- правило «один модуль — один живой документ ТЗ + roadmap для агентов».
